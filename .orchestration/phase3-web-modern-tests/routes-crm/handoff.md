## Mock Pattern (template for future route tests)

The same pattern the `phase2-inventory-tests/routes-inventory` worker
established (and that I followed verbatim):

```ts
const mocks = vi.hoisted(() => ({
  search: { view: "list", status: "all" },  // URL state
  quotes: null as unknown,                    // crm-quotes query data
  forecast: null as unknown,                  // crm-forecast query data
  leads: null as unknown,                     // crm-leads query data
  loading: false,
  fullPath: "/app/crm/",
}));

vi.mock("@tanstack/react-router", () => ({
  createFileRoute: () => (cfg) => ({
    fullPath: mocks.fullPath,
    useSearch: () => mocks.search,
    useParams: () => mocks.params ?? {},
    useNavigate: () => vi.fn(),
    options: cfg,        // exposes validateSearch + component
    update: (u) => u,
  }),
  Link: ({ children, to, ...rest }) => (
    <a data-href={to} href={to} {...rest}>{children}</a>
  ),
  useNavigate: () => vi.fn(),
  notFound: () => { throw new Error("notFound() called"); },
}));

vi.mock("@tanstack/react-query", async (importOriginal) => {
  const actual = await importOriginal();
  return {
    ...actual,
    useQuery: ({ queryKey }) => {
      const key = queryKey[0];
      // route-specific keys + calm empty for child-component keys
      return key === "crm-quotes"  ? { data: mocks.quotes,   isLoading: mocks.loading }
           : key === "crm-forecast"? { data: mocks.forecast, isLoading: mocks.loading }
           : key === "crm-leads"   ? { data: mocks.leads,    isLoading: mocks.loading }
           : { data: null, isLoading: false };
    },
    useQueryClient: () => ({ invalidateQueries: vi.fn() }),
  };
});

vi.mock("../../../lib/api/client", () => ({
  getJson: vi.fn().mockResolvedValue({}),
  postJson: vi.fn().mockResolvedValue({}),
}));

import { Route } from "./index";
// Route.options.validateSearch — pure, just call it
// Route.options.component     — render inside <QueryClientProvider>
```

`vi.hoisted` is the key — it lifts `mocks` above the `vi.mock` factories
so the mocked hooks can read the same shared state the test body
mutates in `beforeEach`.

The `notFound` mock throws an `Error("notFound() called")` so the test
can assert `expect(() => renderRoute()).toThrow(/notFound/)` — a
compact way to pin the "not found" branch without rendering an
error boundary.

## Route Structure Notes (for the next pass)

Both routes are large (636 + 406 lines) and follow the same shape:

1. **Public surface** — only the `Route` export. Sub-components and
   helpers are module-private.
2. **URL state** — `index.tsx` uses `validateSearch` to coerce `view`
   and `status`; we pin this logic with 6 dedicated tests.
3. **Inline sub-components** that are reachable through the rendered
   tree (and therefore tested through it):
   - `index.tsx`: `PageHeader`, `QuoteTable`, `QuoteKanban`,
     `LeadsView`, `EmptyState`, `ForecastTotals`, the `STATUS_TABS`
     filter nav.
   - `$quoteId.tsx`: `QuoteHeader`, `LineTable`, `TotalsBlock`,
     `QuoteMetadata`, `Row`. The right rail also mounts the real
     `AgentActionPanel` and `PricingEvidence` components.
4. **Status filter tabs** in `index.tsx` use `<button>` inside a
   `<nav aria-label="Filter by status">` with `aria-current="page"`
   on the active button — NOT the `role="tab"` pattern that
   `ViewSwitcher` uses. Tests that want the filter tabs MUST scope
   queries to that nav.
5. **`notFound()` wiring** in `$quoteId.tsx` — when the query errors
   or returns null, the route calls `notFound()` from
   `@tanstack/react-router`. We assert the throw with
   `expect(() => renderRoute()).toThrow(/notFound/)`.
6. **Margin classification** in `$quoteId.tsx` (line 277-323) is a
   four-band piece of inline logic (green ≥ 25%, amber ≥ 10%, red
   < 10%, em-dash for null). We pin it with 4 dedicated tests using
   fixed margins 32.5 / 15.0 / 5.0 / null.

## What I Did NOT Test (and Why)

- **The kanban bucket logic** in `index.tsx` (lines 437-451 — the
  `for` loop that pushes each quote into the right column). It's
  inline and only reachable by switching the URL to
  `?view=kanban`, which makes the `KanbanBoard` mount. The phase2
  worker tested this in `inventory/index.test.tsx` via a similar
  approach; I added one smoke test ("renders the kanban view
  without throwing when view=kanban") to ensure the mount path
  doesn't blow up, but did not pin per-column counts because the
  `KanbanBoard` component is opaque from the route test's
  perspective (it renders the column titles and counts internally
  and we don't have stable test-ids for them).
- **The `quotes` filtered-search logic** in `index.tsx` (the
  in-component `useMemo` that filters by title / customerName /
  number). I did not write a UI-driven test for the text search
  input — driving the controlled `useState` through the rendered
  tree would require `@testing-library/user-event` and the search
  input is one of three state sources (URL `status` + URL `view` +
  local `query`). The phase2 worker's strategy of testing the
  filter + sort behaviour through the URL state is what I mirrored.
- **The `LeadCaptureForm` form mechanics** on the leads view right
  rail. The form is non-trivial to drive and there's no value in
  re-testing form mechanics here (it has its own test if needed
  in `web-modern/src/components/lead/LeadCaptureForm.test.tsx` —
  not in scope).
- **The TanStack Router URL update** (`setView`, `setStatus`,
  `onApproved` invalidation). These all flow through the mocked
  `useNavigate` / `useQueryClient` and are tested indirectly via
  the assertion that the route doesn't throw.
