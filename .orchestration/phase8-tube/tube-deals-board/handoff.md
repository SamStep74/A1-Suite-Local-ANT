## Decisions and constraints
- **TubeDealEnvelopeSchema is local to `$dealId.tsx`.** The per-id
  endpoint returns `{ deal }`, but the port-owned
  `TubeListResponseSchema` does not include a `deal` field. I defined
  a local `z.object({ deal: TubeDealSchema.optional() })` in
  `$dealId.tsx` rather than expand the port's schema. If the port
  later adds a `deal` key, this local schema can be deleted.
- **Right-rail panel uses `tubeAgents` directly, not
  `AgentActionPanel`.** The existing
  `components/agent-panel/AgentActionPanel.tsx` reads from
  `lib/agents/registry.ts` which exposes `AGENTS` (salesQuoteAgent,
  inventoryRiskAgent) and does NOT include `tubeAgents`. Using the
  existing panel would have filtered out every tube suggestion. I
  built an inline panel in `$dealId.tsx` that iterates `tubeAgents`,
  filters by `triggers.includes("tube.deal")`, and calls each
  agent's `evaluate()` with a per-deal context. Decisions still hit
  the same `api(path, null, { method, body })` so RBAC, audit, and
  idempotency all stay on the same handler a human would hit.
- **No new registrations in the legacy registry.** A future polish
  could merge `tubeAgents` into `AGENTS` and let
  `AgentActionPanel` accept a `agents` prop; that's out of scope for
  this slice.
- **Drag-and-drop is OUT.** V1 is click-only. Cards link to
  `/app/crm-tube/deals/$dealId`. A "move stage" affordance will
  land with the integrations worker (worker 3/3) or a follow-up.
- **New-deal mutation uses `idempotencyKey: tube-deal-${Date.now()}`.**
  Sufficient for V1; the per-deal stage-move endpoint requires the
  same field, so the pattern is consistent.
